# Release process

## Overview

Pushing a `v*` tag triggers `release.yml`, which builds the desktop binary for each supported platform and uploads the artifacts to a GitHub Release. The landing page is deployed separately via `deploy-web.yml` on every push to `main` that touches `apps/web/` or `packages/shared/`.

---

## Desktop release (`release.yml`)

### Trigger

```bash
git tag v1.2.3
git push origin v1.2.3
```

Any tag matching `v*` triggers the release workflow.

### What the workflow does

1. Runs a matrix build (`macos-latest`, `windows-latest`, `ubuntu-latest`) in parallel.
2. Each runner:
   - Installs Bun and Go 1.23.
   - On Linux, installs WebKit system deps (`libgtk-3-dev`, `libwebkit2gtk-4.1-dev`).
   - Installs the Wails CLI: `go install github.com/wailsapp/wails/v2/cmd/wails@latest`.
   - Runs `bun install --frozen-lockfile` (builds the frontend as part of Turbo's dependency order).
   - Runs `wails build -platform <platform>/amd64` in `apps/desktop/`.
   - Packages the output: `tar -czf "option-tab_<version>_<platform>_amd64.tar.gz" ...`
3. Uploads `*.tar.gz` artifacts to the GitHub Release via `softprops/action-gh-release`.

### Asset naming

The current pipeline produces files of the form:

```
option-tab_<version>_darwin_amd64.tar.gz
option-tab_<version>_windows_amd64.tar.gz
option-tab_<version>_linux_amd64.tar.gz
```

where `<version>` is the tag name with the leading `v` stripped (e.g., tag `v1.2.3` → version `1.2.3`).

## How the landing page resolves download links

`@option-tab/shared` (`packages/shared/src/index.ts`) defines `releaseAssetName(platform, arch, version)` and `downloadUrl(platform, arch, version)`. The landing page imports these functions and passes `APP_VERSION` from `apps/web/lib/download.ts` to construct download URLs.

When a new desktop release is made, update `APP_VERSION` in `apps/web/lib/download.ts` to match the release tag version. This change to `apps/web/` will automatically trigger `deploy-web.yml` on push to `main`, updating the landing page's download links.

---

## Known scaffold limitation

> **The current release pipeline is intentionally minimal.** The `@option-tab/shared` contract defines platform-appropriate asset extensions (`darwin → .dmg`, `windows → .zip`, `linux → .tar.gz`), but the `release.yml` matrix currently packages all three platforms as `.tar.gz`. This means the download URLs generated by the landing page do not yet match the actual release artifacts.

The following items are documented follow-ups (not yet implemented):

| Follow-up | Description |
|-----------|-------------|
| macOS `.dmg` packaging | Package the Wails output as a `.dmg` to match the `@option-tab/shared` contract and macOS user expectations |
| macOS code signing + notarization | Required for Gatekeeper on macOS 10.15+; needs Apple Developer credentials in repository secrets |
| Windows `.zip` packaging | Package the Windows binary as `.zip` to match the contract |
| Windows code signing | Optional but reduces SmartScreen warnings |
| arm64 matrix entries | Add `darwin/arm64` (Apple Silicon) and `linux/arm64` to the build matrix; update `@option-tab/shared` and landing page defaults accordingly |
| Auto-update integration | Integrate a Wails auto-update mechanism pointing at the GitHub Releases API |

Until these follow-ups are implemented, users on macOS and Windows must manually extract the `.tar.gz` archive. The Linux asset matches the contract today.

## Landing page deploy (`deploy-web.yml`)

### Trigger

- Push to `main` that touches `apps/web/**` or `packages/shared/**`.
- Manual dispatch via the GitHub Actions UI.

### What the workflow does

1. Installs Bun and runs `bun install --frozen-lockfile`.
2. Runs `cd apps/web && bun run build`, which produces a static export in `apps/web/out/`.
3. Uploads `apps/web/out/` as a GitHub Pages artifact and deploys it.

The deployed site is available at the repository's GitHub Pages URL. No server-side rendering is involved; the site is a fully static export.

## Versioning convention

- Use [Semantic Versioning](https://semver.org): `vMAJOR.MINOR.PATCH`.
- Tag on `main` after merging the release PR.
- The release PR should bump `APP_VERSION` in `apps/web/lib/download.ts` and update the changelog.
- Commit message for the bump: `chore(release): bump version to X.Y.Z`.
